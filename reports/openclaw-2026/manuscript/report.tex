\subsection{Executive Summary}\label{executive-summary}

In this 24-hour controlled run, the baseline lane executed with a
permissive allow-all decision rule and continued executing post-stop
tool calls after stop commands. It executed \textbf{515 post-stop
calls}, \textbf{497 destructive attempts}, and \textbf{707 sensitive
accesses without approval}. In the governed lane, destructive actions
were held non-executable at \textbf{100\%} under the same matched
workload profile.

Technically, governed evaluation processed \textbf{2,585} lane-specific
tool-call decisions and classified \textbf{1,615} as non-executable
policy violations (\texttt{block} + \texttt{require\_approval}) using
deterministic policy rules. The baseline lane processed \textbf{1,306}
lane-specific calls under \texttt{matched\_rule\_allow\_live} (no
enforceable approval boundary). These totals are observed counts within
each lane for the same scheduled scenario profile, not one-to-one paired
tool calls.

This report is intentionally scoped to one pinned OpenClaw source
snapshot and one controlled 24-hour run. It is not an ecosystem census.
The enforcement mechanism is designed to be portable across agent
runtimes that expose pre-execution tool-call hooks, but numeric outcomes
in this document are case-study-specific to this workload, policy set,
and source pin. All headline values are artifact-backed and
query-reproducible from repository contents.

\subsubsection{Interpreting Discovery vs Runtime
Findings}\label{interpreting-discovery-vs-runtime-findings}

Wrkr in this run is a pre-test discovery and posture scan over
repository/workspace configuration and detected tool inventory. The
high-impact behavior measured elsewhere in this report (delete-email,
public-share, payment approval, restart-service) comes from runtime
tool-call execution traces under workload, not from static repository
metadata alone.

This is expected and is a core result: discovery is necessary for
inventory and baseline posture, but it is insufficient by itself for
runtime action control. Runtime enforcement and decision logging are
required to prevent executable high-risk actions.

\subsection{6) Five Lessons}\label{6-five-lessons}

\begin{enumerate}
\def\labelenumi{\arabic{enumi}.}
\item
  Inventory before scale. Evidence: Pre-test scan produced explicit
  inventory and privilege-budget outputs before workload execution.
  Action implication: Inventory and permission surface should be
  mandatory preconditions for agent deployment.
\item
  Privilege must be enforced at the tool boundary. Evidence:
  Non-executable governed outcomes reached 1,615 in the same matched
  workload profile where baseline-lane actions executed directly. Action
  implication: Instruction-only controls without enforceable policy at
  execution time are not a sufficient boundary in this harness.
\item
  Evidence infrastructure has to exist before incidents. Evidence:
  Governed lane produced verifiable decision traces at 99.96\% coverage.
  Action implication: Incident response requires artifact-backed
  decision history, not reconstructed narratives.
\item
  Approval flows must be explicit and enforceable. Evidence: 337
  governed write-class actions moved to \texttt{require\_approval}
  instead of executing. Action implication: Approval semantics should be
  machine-enforced, not advisory.
\item
  Stop semantics require enforceable runtime controls. Evidence:
  Baseline lane showed executable post-stop actions (\texttt{515/515}),
  while governed lane enforced non-executable outcomes after stop under
  the same matched workload profile. Action implication: Stop signals
  alone are not enough; systems need enforceable deny/approval/allow
  controls at tool execution time.
\end{enumerate}

\subsubsection{What This Means for
Organizations}\label{what-this-means-for-organizations}

If your organization gives AI agents tool access to email, file sharing,
financial operations, or infrastructure actions, this run identifies a
failure mode worth testing in your own environment under controlled
conditions. The operational question is whether tool-boundary
enforcement exists at execution time, or whether controls rely mainly on
prompt instruction and best-effort model compliance.

As context-only industry framing, this runtime control gap is
directionally consistent with broader third-party and supply-chain risk
pressure documented in external threat-intelligence reporting, including
IBM X-Force analyses logged in
\texttt{citations/threat-context-sources.md}.

\subsection{Limitations}\label{limitations}

\begin{itemize}
\tightlist
\item
  This report covers one pinned OpenClaw source snapshot and one
  canonical 24-hour run.
\item
  The workload profile is controlled and scenario-based; it is not a
  census of all production behaviors.
\item
  No confidence intervals are reported; run-to-run variance is not
  estimated in this manuscript.
\item
  This manuscript does not present comparative baselines against
  alternative governance frameworks or prompt-only control experiments.
\item
  Policy set \texttt{openclaw-research-v1.yaml} was authored with
  knowledge of core5 scenario classes.
\item
  \texttt{secrets\_handling} governed non-executable rate is 20\% in
  this run (\texttt{90/450} non-executable, \texttt{360/450} allowed),
  indicating policy tuning is still required for sensitive-export
  pathways.
\item
  External incident reporting is used only as context and is not treated
  as numeric evidence for claims.
\end{itemize}

\subsection{Threats to Validity}\label{threats-to-validity}

\begin{itemize}
\tightlist
\item
  Workload-shape bias: fixed scenario scheduling may over- or
  under-represent real user sequences.
\item
  Classification bias: sensitive/destructive labels are deterministic
  but still depend on schema mappings.
\item
  Environment bias: isolated lab controls differ from live enterprise
  integration environments.
\item
  External-validity risk: the control mechanism is portable in
  principle, but metric magnitudes can shift under different runtimes,
  tools, policy mappings, and workload mixtures.
\item
  Baseline semantics: the baseline lane uses a permissive allow-all
  decision rule (\texttt{matched\_rule\_allow\_live}) in this harness.
\item
  Stop-metric semantics: ungoverned stop behavior is measured from
  post-stop executable call traces rather than explicit stop-signal
  event counters.
\item
  Stop-latency artifact: \texttt{stop\_to\_halt\_p95\_sec\ =\ 0}
  reflects synchronous harness behavior, not end-to-end distributed
  production latency.
\item
  Tooling drift: upstream changes in OpenClaw, Wrkr, Gait, or
  dependencies can alter observed distributions.
\end{itemize}

\subsection{Residual Risk}\label{residual-risk}

\begin{itemize}
\tightlist
\item
  Even with non-executable enforcement, policy gaps can allow low-risk
  or safe-read pathways that may still expose sensitive context.
\item
  Governance effectiveness depends on policy quality and endpoint
  classification completeness.
\item
  Approval mechanisms reduce immediate execution risk but do not replace
  human review quality.
\end{itemize}
